反逆向是指软件开发者通过各种技术手段防止程序被逆向分析、调试或破解的过程。

本文对一些反逆向手段进行总结。

<!--more-->

- 反调试
- 花指令
- 加壳
- 虚拟机保护
- 利用 Windows 异常处理反调试

### 反调试

1. **TLS 回调**

TLS 在 main 函数之前执行，也早于全局变量的初始化，可以执行一些反调试代码，也可以初始化一些变量。

在 Windows 系统中，每个 TLS 段可以注册多个回调函数，可以在进程启动/结束、线程创建/结束时调用

2. **自调试技术**

多数操作系统限制**一个进程同一时间只能被一个调试器附加**，程序自己调试自己，抢占调试端口，外部调试器就无法附加

Linux 使用 `ptrace` 命令附加
```c
if (ptrace(PTRACE_TRACEME, 0, NULL, NULL) == -1) {
    // 附加失败，可能已被调试
    exit(1);
}
```


3. **调试器检查**

检查是否被调试，常用作反调试手段

Windows 内部API ：`IsDebuggerPresent()` 、`CheckRemoteDebuggerPresent()` 远程调试检测




### 花指令

花指令的目的是干扰反汇编和反编译

先了解两种反汇编算法：

1. **线性扫描**

	扫描二进制，按字节顺序进行指令解析

	但如果数据与指令混在一起，可能将数据错误识别成指令，即无法识别代码中内嵌的数据。

2. **递归下降**

	从入口出发，分析控制流，反汇编可能执行到的字节

	> 间接跳转利用寄存器进行跳转，可能导致覆盖不全

	可能被虚假控制流干扰：

	- 具有相同目标的跳转指令
	- 始终执行的条件跳转指令（不透明谓词）

再介绍一些干扰反编译的手段：

- ret 指令滥用：IDA 分析函数时，碰到 ret 指令就认为当前函数分析结束。将指定地址压栈、 ret 操作就能干扰 IDA 的反编译 F5，不会分析 ret 后面的代码
- 破坏堆栈平衡：IDA 分析函数时回计算堆栈指针的变化，如果不平衡就不能 F5

- 利用异常处理隐藏代码：IDA 不能很好处理 try-catch ，反编译时可能无法显示错误处理逻辑

- 利用 ROP 技术隐藏代码



### 壳

#### upx
```bash
sudo apt update
sudo apt install upx-ucl
```

- 压缩：通过压缩和加壳技术减小文件体积
- 加壳：将可执行程序的外部再包一层外壳程序，并在运行时使用外壳程序解压实际的程序，然后加载到内存运行。

可使用 **Die (Detect  it easy)** 检测文件类型，判定是否加壳

加壳后的程序应包含：

- PE 头部：保留原始 PE 文件的 DOS 头、NT 头和节表（Section Table），但可能被修改以适应压缩后的结构。

- UPX 区段：UPX 通常将文件分为几个自定义区段（如 `UPX0`、`UPX1`、`UPX2`），用于存储压缩数据、解压 stub 和其他元数据。

- UPX 头部：UPX 在文件中嵌入一个特定的头部结构，包含压缩算法标识、文件长度等信息，`UPX! `是其中的一个标志性字符串。

- 解压 stub：一段解压代码，负责在运行时将压缩的程序还原到内存中。

- 压缩数据：原始程序的代码和数据经过压缩后存储在文件中。

UPX 的加壳过程会重构 PE 文件的节表和内容，但保留 PE 格式的基本结构以确保文件仍可被操作系统加载。

对upx 加壳过的程序：逆向时需要使用 `upx -d` 脱壳：

脱壳过程中 upx 特征区段很重要，逆向时若无法脱壳需要考虑恢复特征区段

- 修改区段名：`UPX0`  `UPX1`  `UPX2` 可以改成其他名字，`upx -d` 将无法识别文件结构，导致脱壳失败。
- 恢复UPX头：恢复 `UPX!`

```
版本号 + 00 55 50 58 21 0D xx 02 08
```



如果仍然无法脱壳，可能需要手动脱壳


#### 手动脱壳

1. 寻找入口点

	使用 x64dbg 打开直接运行一步，会停在壳的入口点（pushad，将所有寄存器的值压栈，保护现场），接下来是解压内部程序，完成后恢复现场（popad），程序控制权交给内部程序

	关键的就是找到恢复现场后的内部程序入口点，在壳的入口点对栈上的寄存器下硬件断点，弹出时说明正在恢复现场，恢复现场后可能还有一些修复操作，这时候就要根据经验找到程序入口点了，一般就在一个较远的 jmp 目标处

2. dump 内存到文件，修复文件

	使用 x64dbg 自带的 Scylla 进行脱壳，然后 IAT Autosearch -> Get Impoets -> Dump

	此时内部程序正在入口处，从内存中 dump 出程序到文件，这个程序大概率不能运行，但是已经可以静态分析了。



### 虚拟机

虚拟机 VM 保护：将原始机器代码（如x86指令）转换（或“编译”）为只能在自定义的虚拟机上执行的字节码（Bytecode）。

VM 保护创建了一个私有的、独特的“处理器”和“指令集”（通常称为 VM Architecture 或 ISA ），来执行被保护的代码。

- vm_start：虚拟机入口函数，初始化虚拟机环境
- vm_dispatcher：调度器，解释 opcode，并选择对应的 handle 函数执行，执行完会跳回这里，形成循环
- opcode：程序可执行代码转换成的操作码

典型的代码结构就是 while + switch 语句和 while 多条 if 语句



#### 静态分析

提取 opcode 、handle 重建虚拟机指令集，然后分析加密执行流，推导或爆破出明文

如果执行流加密操作可逆，直接倒推，如果不可逆考虑爆破

- Z3 约束求解
- 符号执行
- 重建虚拟机



#### 动态分析

通过插桩 VM dispatcher / handle 函数，实时观察每一次 opcode 执行时的寄存器/内存状态，得到虚拟汇编执行流

接下来方法同静态分析

插桩具体方法：

- 断点+脚本：IDA + Python
- 手动 Hook：手动改汇编，jmp 到 log 函数
- Frida 动态注入：动态注入 + JS/Python
- Intel Pin / DynamoRIO：动态二进制插桩框架





### 异常

1.硬件异常：CPU 检测到的异常

- `fault`：保存上下文，调用错误处理函数，然后重新执行。如 Page Fault
- `trap`：保存上下文，处理异常后，直接执行导致异常的指令的下一条指令。如断点，指令`int 3` ，会触发中断 interrupt 到调试器
- `abort`：中止异常，不会恢复执行，会强制性退出

2.软件异常：程序显式触发的异常

- 系统定义的异常（如 `RaiseException()`、`int 3`、`0x00` 除零）
- 用户自定义异常（`try-catch` 结构或 `__try/__except`）

---

Windows 异常处理：分为 R0 内核异常处理和 R3 用户异常处理

> 在 Windows 系统中，硬件异常和中断被不加区分的存放在了一个向量表中，也就是 IDT（interruption descriptor table）



#### R3 异常处理

异常处理器处理流程：

- 交给调试器（**First Chance** 第一次机会）
- 执行 **VEH**（Vectored Exception Handling，向量化异常处理）
- 执行 **SEH**（Structured Exception Handling，结构化异常处理）
- 再次交给调试器（**Second Chance** 第二次机会）
- 执行 `TopLevelEH`（`UnhandledExceptionFilter` / `SetUnhandledExceptionFilter` 设置的过滤器）
- 调用异常端口通知 `csrss.exe`，最终终止进程（或执行默认的错误报告）

异常的传递过程：调试器 -> VEH -> SEH -> 调试器 -> UEH -> VCH 

> UEH：UnhandledExceptionFilter / TopLevelEH：只有无调试器或调试器放弃时才会调用注册的 TopLevel Filter。
> VCH：Vectored Continue Handler（向量化继续处理器）：在某个 handler（VEH 或 SEH）返回 EXCEPTION_CONTINUE_EXECUTION（决定处理这个异常并继续执行）之后，被调用来做“继续阶段”的清理或额外处理。



##### SEH 反调试

SEH 是 Windows 经典的栈式异常处理机制，每个线程维护一条 SEH 链

自定义 SEH handler，然后故意触发异常

- 有调试器时：调试器先拿到 First Chance，handler 可能被跳过或行为异常。
- 无调试器时：直接进入 SEH handler。



##### VEH 反调试

VEH 是 Windows XP 引入的全局异常处理机制（不依赖线程栈，更难被绕过）。

VEH 反调试：注册 VEH → 触发异常 → 在 handler 中判断 ExceptionCode 和上下文（ContextRecord->Eip / Rip）。

调试器会拦截部分异常（如 STATUS_BREAKPOINT），导致 VEH 行为与无调试器时不同。



##### TopLevelExceptionFilter 反调试

1. 先调用 SetUnhandledExceptionFilter 设置自己的过滤器。
2. 触发一个必然未处理的异常。

- 有调试器时：调试器会接管 Second Chance，UnhandledExceptionFilter() 不会调用 SetUnhandledExceptionFilter() 设置的自定义过滤器。
- 没有调试器附加时（正常运行）：UnhandledExceptionFilter() 会调用之前用 SetUnhandledExceptionFilter() 设置的自定义过滤器函数（TopLevelEH）。
